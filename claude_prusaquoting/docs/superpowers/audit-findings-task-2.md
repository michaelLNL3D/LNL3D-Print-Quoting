# Audit Findings: Batch Mode

| ID | Title | Category | Severity | Lines | Description | Proposed Fix |
|----|-------|----------|----------|-------|-------------|-------------|
| 2.1a | Settings snapshot taken lazily, not on add | Design / UX | Medium | 2119-2126 | `addFilesToQueue()` pushes items with no `settings` field (`{file, status:'pending', result:null}`). Settings are only captured when the user explicitly clicks a different queue item (via `selectQueueItem`, line 1949-1950) or at batch-run time (line 2282-2284 saves the currently-selected item). If a user adds 3 files, changes settings, then clicks "Quote All" without ever clicking on individual queue items, only the currently-selected item (if any) gets its settings saved. The other items use `getFormSettings()` at slice time (line 2091 fallback in `buildFormData`), which reads whatever the form shows at that moment. This means all non-selected items effectively share the form's settings at the time each is sliced -- which is the same settings since the form doesn't change during batch processing. This is arguably correct "shared settings" behavior, but the gear icon (line 2139) only appears when `item.settings` is truthy, giving a false impression that items without the badge have no custom settings when they actually just inherit the current form. | Either snapshot settings for all items at batch-run time (iterate `fileQueue` and set `.settings = getFormSettings()` for any item without custom settings), or document the "shared settings" behavior more explicitly. |
| 2.1b | Selecting queue item without prior settings does not update form | OK | Info | 1947-1956 | `selectQueueItem(i)` saves the current form into the previously selected item (line 1949-1950), then checks `if (item.settings)` before calling `applyFormSettings`. If the newly selected item has no saved settings, the form retains whatever was showing before. This is intentional -- the item inherits the current form state. No bug, but could confuse users who expect clicking a "blank" item to reset the form. | Consider showing a visual indicator on the form when viewing inherited vs custom settings. |
| 2.1c | Settings drift between form and queued snapshot | Bug | Low | 1947-1956 | After clicking item A (saving its settings), the user can modify the form without clicking back to item A. Those modifications are not captured in item A's snapshot until the user clicks another queue item or runs the batch. If the user directly hits "Quote All", line 2282-2284 only saves the *currently selected* item. So if selectedQueueIdx is item B, item A's snapshot is stale. However, since `buildFormData` uses `item.settings` (which was the snapshot from when the user clicked away from A), this is the expected per-file customization behavior. The real risk: if user clicks A, tweaks settings, then hits Quote All without clicking elsewhere -- those tweaks go to A's live snapshot (line 2283), which is correct. | Low risk; current behavior is mostly correct. Could add a "save current settings" prompt or auto-save on any form change via event listeners. |
| 2.2a | Skip-done works correctly in batch loop | OK | Info | 2393-2395 | `_runBatchSlice()` iterates all `fileQueue` items and checks `if (item.status === 'done') continue;` at line 2395. Already-completed items are properly skipped. The `total` count at line 2390 also only counts non-done items. | No change needed. |
| 2.2b | No explicit "re-quote single item" mechanism | Design | Low | 2177-2180, 2385-2418 | There is no button or function to force re-quote a single already-done item in batch mode. The only way to re-quote is to run the entire batch again, but done items are skipped. A user would need to remove the item and re-add it, losing its per-file settings. In single-file mode (line 2180), `runQuote` resets `fileQueue[0].status = 'pending'` before quoting, so single-file re-quote works. | Add a per-item "Re-quote" button in the queue list or batch results table that resets `item.status = 'pending'` and optionally triggers a single-file slice. |
| 2.2c | Re-running batch after all items done does nothing visible | Bug | Low | 2390-2391 | If all items are `done` and user clicks "Quote All" again, `total` is 0, the loop body never executes, but the function still calls `showBatchResults()` at line 2423. No progress panel is shown and it appears to do nothing besides re-rendering the existing results table. Not harmful but confusing -- the button appears active but has no effect. | Either disable the button when all items are done, or reset all items to pending before re-running, or show a message like "All files already quoted." |
| 2.3a | `deleteQueueItem` index handling is correct | OK | Info | 1958-1980 | When deleting item at index `i`: if `selectedQueueIdx === i`, it resets to -1; if `selectedQueueIdx > i`, it decrements. This correctly maintains the selected index after splice. If queue becomes empty, `clearQueue()` is called. | No change needed. |
| 2.3b | `deleteQueueItem` unconditionally enables run button | Bug | Low | 1965-1967 | After deleting, line 1966 sets `btn.disabled = false` without checking if a printer is selected. `updateRunBtn()` is not called, so the "Generate Quote" / "Quote All" button may be enabled even though no printer is selected. | Replace `btn.disabled = false` with a call to `updateRunBtn()`. |
| 2.3c | `clearQueue` does not fully reset results panel | Bug | Medium | 2147-2157 | `clearQueue()` resets the queue array, selected file, and index, hides the preflight panel, and updates the drop zone. However, it does NOT: (1) hide the `#results` panel, (2) hide the `#batch-results` panel, (3) hide the `#error-box`, (4) reset `window._lastQuote`, (5) hide the `#rc-tabs`. Stale results from a previous batch remain visible after clearing the queue. | Add cleanup: `document.getElementById('results').style.display='none'; document.getElementById('batch-results').style.display='none'; document.getElementById('error-box').style.display='none'; document.getElementById('rc-tabs').style.display='none'; document.getElementById('placeholder').style.display='flex'; window._lastQuote=null;` |
| 2.3d | `deleteQueueItem` transitions from batch to single-mode results correctly | OK | Info | 1968-1979 | When deleting leaves >1 items, `showBatchResults()` is called to refresh the table. When exactly 1 done item remains, `showResults(fileQueue[0].result)` is called to show single-mode view. When 1 pending/error item remains, placeholder is shown. All three branches are handled. | No change needed. |
| 2.4a | Batch totals use numeric addition via `minsFromStr`, not string concat | OK | Info | 2427-2444 | `totalMins` is accumulated via `minsFromStr(d.time)` which parses the time string into minutes (line 2436). `totalG` accumulates `d.weight_g` numerically (line 2438). `totalCost` accumulates `d.quote_price` or `d.subtotal` numerically (line 2440). The totals row at line 2444 uses `minsToStr(totalMins)` to format back to string. All correct. | No change needed. |
| 2.4b | Row clicks correctly open modal with right index | OK | Info | 2442, 2455-2497 | Each table row has `onclick="openQuoteModal(${i})"` where `i` is the `fileQueue` index from the `.map()` callback. `openQuoteModal(i)` accesses `fileQueue[i]` and reads `item.result`. The index is correct because `showBatchResults` iterates `fileQueue` with its natural indices. | No change needed. |
| 2.4c | Error rows in batch table show error text but are clickable | OK | Info | 2433-2434 | If `d.error` is present, the row shows the error text in a colspan cell and remains clickable (opens modal). `openQuoteModal` at line 2457 has a guard `if (!item?.result) return;` but error items DO have a result (with `.error` field), so the modal opens. The modal then shows "undefined" or incomplete data since fields like `time`, `weight_g` are missing on error results. | Consider showing a dedicated error view in the modal, or skipping the modal for error items. |
| 2.4d | Batch table not hidden when switching to Health tab | OK | Info | 2048-2058 | `switchRcTab('health')` correctly hides `batch-results` by only showing it when `showQuote && !preflighting && fileQueue.length > 1 && ...` (line 2054). When tab is 'health', `showQuote` is false, so batch-results is hidden. | No change needed. |
| 2.5a | Batch processing continues after individual file failure | OK | Info | 2405-2417 | The `try/catch` in `_runBatchSlice` wraps each individual file's fetch. On HTTP error (`!res.ok`), `item.status` is set to `'error'` and `item.result` stores the error data (line 2409-2410). On network exception, the catch block sets `item.status = 'error'` with a synthetic error result (line 2413-2414). The `for` loop continues to the next file in both cases. | No change needed. |
| 2.5b | Failed file status is visible in queue and batch table | OK | Info | 2137, 2140 | The queue list renders with CSS class `st-${item.status}`, and displays the status text. Error items show `st-error` class. In the batch results table (line 2433-2434), error rows show the error message. | No change needed. |
| 2.5c | Error items included in batch total count but not in cost/time totals | OK | Info | 2433-2443 | Error rows skip the `totalMins`/`totalG`/`totalCost` accumulation because they hit the early return `if (!d || d.error)` at line 2433. But they ARE counted in the total row label `TOTAL (${fileQueue.length})` at line 2444. This means the total row says e.g. "TOTAL (5)" but the summed values only reflect 4 successful files. | Consider showing "TOTAL (4/5 succeeded)" or separating the count to avoid confusion. Minor UX issue. |
| 2.6a | Single-to-batch transition on file add | OK | Info | 2119-2126 | Adding a second file calls `addFilesToQueue`, which updates button text to "Quote All" (line 2125) if `fileQueue.length > 1`. The queue list renders (hidden when <=1 file, shown when >1). No results are cleared at this point -- any existing single-file results remain visible until the next quote run. | This is acceptable; results are from the previous run and will be replaced. |
| 2.6b | `_lastQuote` from single mode leaks into batch mode | Bug | Low | 2272, 2652 | When `runBatch()` starts, it sets `window._lastQuote = null` (line 2272), which is correct. However, `showResults()` at line 2652 sets `window._lastQuote = d`, and this is called from `deleteQueueItem` when reducing to 1 done item (line 1973). If the user then adds more files (going back to batch mode), `_lastQuote` still holds the old single result. `switchRcTab('quote')` at line 2053 checks `window._lastQuote` to decide whether to show the single results panel. This could cause the single-file results panel to flash briefly or appear alongside the batch table if tab switching happens before the next quote run. | Clear `_lastQuote` in `addFilesToQueue` when transitioning from 1 to 2+ files, or in `runBatch`. Already cleared in `runBatch` so risk is limited to tab-switching before re-quoting. |
| 2.6c | `runQuote` correctly routes single vs batch | OK | Info | 2177-2178 | `runQuote()` checks `if (fileQueue.length > 1)` and delegates to `runBatch()`. With exactly 1 file, it falls through to single-file flow. With 0 files (shouldn't happen since button is disabled), it returns at line 2179. | No change needed. |
| 2.6d | `updateRunBtn` does not account for batch mode | Bug | Low | 2886-2891 | `updateRunBtn()` at line 2889 checks `!selectedFile` to decide if the button should be disabled. In batch mode, `selectedFile` is set to `fileQueue[0]?.file` (line 2121, 1963). If `fileQueue[0]` exists but no printer is selected, the button correctly stays disabled. However, `updateRunBtn` never updates the button *text* -- it doesn't distinguish between "Generate Quote" and "Quote All". The text is only set in `addFilesToQueue` (line 2125) and `deleteQueueItem` (line 1967). If `updateRunBtn` is called from `loadProfiles` (line 2830) after the queue is populated, the button text could revert to the wrong label. | Have `updateRunBtn` also set `btn.textContent` based on `fileQueue.length`. |
