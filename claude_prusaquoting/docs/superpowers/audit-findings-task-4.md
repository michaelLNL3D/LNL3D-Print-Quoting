# Audit Findings: State & Edge Cases

| ID | Title | Category | Severity | Lines | Description | Proposed Fix |
|----|-------|----------|----------|-------|-------------|-------------|
| 4.1a | Progress entry leaks on page refresh mid-quote | Bug | Medium | 42, 2018-2023 | When a user refreshes mid-quote, the JS `stopProgressPoll()` never fires, so the `DELETE /api/progress/<pid>` call is never made. The old `pid` entry stays in the `_progress` dict permanently. The `_progress` dict has no size cap (unlike `_jobs` at 50 or `_error_log` at 100), so repeated refreshes during slicing cause unbounded memory growth. | Add a TTL or max-size cap to `_progress`. For example, evict entries older than 5 minutes on each `_emit()` call, or cap at 100 entries like `_error_log`. |
| 4.1b | Slicer subprocess orphaned on page refresh | Bug | Medium | 87-91, 652, 900 | `run_slicer()` uses `subprocess.run()` (blocking), and `api_quote()` runs synchronously in a Flask thread. When the user refreshes, the HTTP connection is dropped but Flask's threaded server continues executing the request handler to completion, including the blocking `subprocess.run()`. The slicer process is NOT orphaned -- it runs to completion -- but the results are discarded (the response has nowhere to go). This wastes CPU/time but is not a resource leak. | Acceptable for a single-user tool. For multi-user, consider `subprocess.Popen` with cancellation tokens. |
| 4.1c | New page after refresh shows no stale progress | OK | Info | 1797, 2007-2015 | On refresh, `init()` does not attempt to restore any `pid` or poll any old progress. The progress panel starts hidden, and `_activePid` is `null`. The new page is clean. No stale progress is displayed. | No fix needed. |
| 4.2a | GCode download after server restart returns unhelpful error | Bug | Low | 1211-1236 | After restart, `_jobs` is empty. `api_export_gcode()` falls back to `_last_job` (also empty), finds no `gcode_paths`, and returns `{"error": "No gcode -- run a quote first"}` with HTTP 400. The error message is clear enough, but the GCode files may still exist on disk in `/tmp/prusaquoting_exports/` -- they are never matched back to job IDs. The UI still shows a download button from the pre-restart quote (stored in JS `item.result.job_id`). Clicking it gives the 400 error. | Error message is adequate. Optionally, persist `_jobs` to disk (e.g., a small JSON file in `EXPORT_DIR`). |
| 4.2b | 3MF export after server restart returns clear error | OK | Info | 1164-1170 | `_last_job` is empty after restart. `api_export_3mf()` returns `{"error": "No job to export -- run a quote first"}` with HTTP 400. The error is clear and the JS `download3mf()` function (line 2706-2709) catches it and shows it via `showError()`. | No fix needed. |
| 4.2c | All in-memory state lost on restart | Design | Low | 35-42 | `_jobs`, `_last_job`, `_progress`, and `_error_log` are all pure in-memory dicts/lists. Server restart loses everything. This is documented as intentional ("single-user tool") but could surprise users who restart after a long batch. | Acceptable for scope. Document in UI or consider persisting `_jobs` mapping. |
| 4.3a | Double-click can fire duplicate quote requests | Bug | Medium | 2177-2183, 2236-2238 | `runQuote()` sets `btn.disabled = true` at line 2183, but only AFTER first checking `fileQueue.length` and `selectedFile`. There is no mutex or request-in-flight guard. A very fast double-click could enter `runQuote()` twice before the first call reaches line 2183. Both calls would proceed to `_doSingleQuote()`, firing two `/api/quote` requests. Each would overwrite `_last_job` (line 886-898) and create separate `job_id` entries. The second response would overwrite the first in the UI. | Add an `_isQuoting` boolean guard at the top of `runQuote()` that returns immediately if already true. Set it false in the finally block of `_doSingleQuote()`. |
| 4.3b | Changing printer dropdown during slicing has no effect on in-flight request | OK | Info | 854-873, 2250-2251 | Form values are captured into `FormData` by `buildFormData()` at the moment the fetch is issued (line 2251). Changing the dropdown afterward does not affect the in-flight request. The server uses only the values from the submitted form. | No fix needed -- correct behavior. |
| 4.3c | Batch mode allows re-quoting already-done items | Design | Low | 2390-2395 | `_runBatchSlice()` skips items with `status === 'done'`, but there is no lock preventing the user from clicking "Quote All" again while a batch is running. The button IS disabled (line 2387), but if somehow re-enabled (e.g., via DevTools), a second batch could run concurrently. | Low risk for a single-user tool. The button disable is sufficient protection. |
| 4.4a | Empty printer list shows placeholder but form remains interactive | Bug | Low | 1797-1811, 1886-1889 | If PrusaSlicer has no profiles (or `get_printer_profiles()` returns empty), the dropdown shows only `"-- select printer --"` with no selectable options. `updateRunBtn()` checks `!!document.getElementById('printer-sel').value` which is empty string (falsy), so the "Generate Quote" button stays disabled. The user cannot submit. However, no guidance text explains WHY no printers appear or how to fix it. | Add a visible notice when the printer list is empty, e.g., "No printer profiles found. Ensure PrusaSlicer is installed with at least one printer configured." |
| 4.4b | Server unreachable during init shows dropdown message | OK | Info | 1808-1810 | If the `/api/printers` fetch fails, the catch block sets the dropdown to `"-- server unreachable --"`. The button stays disabled. This is reasonable feedback. | No fix needed. |
| 4.5a | Missing PrusaSlicer binary causes unhandled FileNotFoundError | Bug | High | 87-88, 96 | `run_slicer()` calls `subprocess.run([PRUSASLICER] + args)` with no existence check. If the binary does not exist, Python raises `FileNotFoundError`. This is NOT caught in `run_slicer()` itself. At startup, `get_printer_profiles()` is called indirectly when the UI loads printers via `/api/printers` (line 672-674). The `FileNotFoundError` propagates up. The global `handle_unhandled_exception` (line 74-82) catches it and returns JSON `{"error": "...FileNotFoundError..."}`, which is functional but unhelpful -- the user sees a cryptic error. | Add an existence check at the top of `run_slicer()`: `if not os.path.exists(PRUSASLICER): raise RuntimeError(f"PrusaSlicer not found at {PRUSASLICER}. Set PRUSASLICER_PATH env var.")`. Or check once at startup. |
| 4.5b | Missing OrcaSlicer binary causes auto-orient to fail silently with poor feedback | Bug | Medium | 90-91, 242-271 | `run_orca()` similarly has no existence check. When OrcaSlicer is missing, `subprocess.run()` raises `FileNotFoundError`. In `auto_orient()`, this exception is NOT caught -- it propagates up to the `api_quote()` handler. The global exception handler (line 74-82) catches it and returns a 500 error, killing the entire quote rather than gracefully skipping orientation. The function's docstring says "Falls back to input_path on failure" but only handles the case where the subprocess runs but produces no output (line 270-271), not the case where it cannot run at all. | Wrap the `run_orca()` call in `auto_orient()` with a try/except that catches `FileNotFoundError` (and `OSError`) and falls back to `(input_path, "orient skipped (OrcaSlicer not found)")`. |
| 4.6a | No cleanup of old GCode files in EXPORT_DIR | Bug | Medium | 30-32, 1013 | GCode files are written to `/tmp/prusaquoting_exports/` with names like `{job_id}_piece_{idx}.gcode`. Each quote creates new files. Old files are never deleted -- not at startup, not on a schedule, not when `_jobs` evicts old entries (line 1120-1121). Over time this directory grows unboundedly. On macOS `/tmp` is cleared on reboot, but on Linux or Docker deployments it may persist. | When evicting from `_jobs` (line 1120-1121), also delete the associated gcode files. Add a startup sweep that removes files older than 24 hours. |
| 4.6b | No cleanup of `last_input*` / `last_work*` files | Bug | Low | 883, 993-994 | Each quote overwrites `last_input{suffix}` and `last_work{suffix}` in `EXPORT_DIR`. However, if the file extension changes between quotes (e.g., first quote is `.stl`, second is `.3mf`), the old `last_input.stl` file remains alongside the new `last_input.3mf`. These are small but accumulate. | Delete any existing `last_input.*` and `last_work.*` files before writing the new ones. |
| 4.6c | Failed slicing leaves partial GCode files | OK | Info | 637, 653 | `slice_piece()` checks `os.path.exists(out_gcode)` after slicing (line 653). If slicing fails, the gcode file typically does not exist (PrusaSlicer does not write partial output on error). The persistent copy (line 658-660) is only made if slicing succeeds. No partial files leak. | No fix needed. |
| 4.7a | `_progress` dict has no size cap and leaks on errors | Bug | Medium | 42, 44-54, 736-743 | `_progress` uses `setdefault` to create entries but has no eviction policy. The only deletion path is the JS `DELETE /api/progress/<pid>` call in `stopProgressPoll()` (line 2022), which itself uses a 3-second `setTimeout`. If the browser tab is closed, refreshed, or the JS errors out before `stopProgressPoll()` runs, the entry persists forever. Unlike `_jobs` (50 cap) and `_error_log` (100 cap), `_progress` is completely uncapped. | Add a cap (e.g., 200 entries) with FIFO eviction in `_emit()`, or add timestamp tracking and evict entries older than 10 minutes on each access. |
| 4.7b | `_jobs` 50-cap is correctly enforced | OK | Info | 1119-1121 | After inserting the new job, `_jobs` checks `len(_jobs) > 50` and deletes the oldest entry (first key in insertion order, since Python 3.7+ dicts are ordered). This works correctly. However, the associated GCode files on disk are NOT deleted (see 4.6a). | Cap enforcement is correct; disk cleanup is the separate issue in 4.6a. |
| 4.7c | `_error_log` 100-cap is correctly enforced | OK | Info | 56-66 | `_log_error()` appends and then checks `len > 100`, popping the oldest entry. This works correctly. | No fix needed. |
| 4.8a | Two tabs overwrite each other's `_last_job` | Bug | High | 35, 886-898, 1164-1198 | `_last_job` is a single global dict. When tab B runs a quote, it overwrites `_last_job` at line 886-898. If tab A then clicks "Download 3MF" or "Export INI", it gets tab B's data. There is no per-session or per-job scoping for `_last_job`. The 3MF export (line 1164-1198) and INI export (line 1201-1208) both read from this single global. The `input_path` and `work_path` files are also overwritten (lines 883, 993), so tab A's model file is physically replaced on disk. | Scope `_last_job` by `job_id`. Store the export metadata in `_jobs[job_id]` instead of a separate global. Have the 3MF/INI export endpoints accept a `job_id` parameter. |
| 4.8b | Progress polling is correctly scoped by unique `pid` | OK | Info | 1982, 2007-2015, 736-738 | Each quote generates a unique `pid` via `makePid()` (line 1982: `Math.random().toString(36).slice(2,12)`). Each tab polls only its own `pid`. Tabs cannot see each other's progress. The 10-character base-36 random string provides ~5.2 * 10^15 combinations, which is sufficient for collision avoidance. | No fix needed. |
| 4.8c | `job_id` generation uses UUID4 truncated to 8 hex chars | OK | Low | 1004-1005 | `job_id = str(_uuid.uuid4())[:8]` gives 8 hex characters = ~4.3 billion combinations. For a single-user tool with at most 50 active jobs, collision probability is negligible. However, in a theoretical multi-user scenario with high throughput, 8 hex chars could eventually collide. | Acceptable for scope. For multi-user, use full UUID or longer prefix. |
| 4.8d | GCode download correctly uses `job_id` parameter | OK | Info | 1211-1216 | `api_export_gcode()` accepts `?job_id=xxx` and looks it up in `_jobs`. If not found, it falls back to `_last_job` (which has the concurrent-tab issue from 4.8a). The batch UI correctly passes `job_id` in download links (line 2134-2135), so batch GCode downloads are per-job and safe across tabs. Single-file mode does not pass `job_id` for GCode download, relying on `_last_job`. | Have single-file mode also use `job_id` for GCode export links rather than relying on `_last_job` fallback. |
