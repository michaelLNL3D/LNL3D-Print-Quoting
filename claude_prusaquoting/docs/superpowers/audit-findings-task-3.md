# Audit Findings: UI/UX Friction

| ID | Title | Category | Severity | Lines | Description | Proposed Fix |
|----|-------|----------|----------|-------|-------------|-------------|
| 3.1a | Drop zone has dragover visual feedback | UX Friction | OK | 1297, 1867 | The `.dragover` class is added on `dragover` and removed on `dragleave`/`drop`, changing border color and background. This works correctly. | None needed. |
| 3.1b | Non-model files silently ignored on drop | UX Friction | Low | 1871 | When files are dropped, a regex filter keeps only `.stl/.obj/.3mf/.amf/.step/.stp` files. If all dropped files are non-matching, nothing happens -- no feedback tells the user their files were rejected. | Show a brief toast or flash message like "Unsupported file type" when all dropped files are filtered out. |
| 3.1c | Non-file drag content (text, URLs) causes no-op without feedback | UX Friction | Low | 1869-1872 | Dragging text or URLs onto the drop zone triggers `preventDefault()` and removes `dragover` class, but `e.dataTransfer.files` is empty so the filter produces an empty array. The drop zone briefly highlights then does nothing. | Check `e.dataTransfer.files.length === 0` and show a message like "Please drop model files." |
| 3.1d | File input `accept` does not match drop filter exactly | UX Friction | Low | 1556, 1871 | The `<input>` `accept` attribute lists `.stl,.obj,.3mf,.amf,.step,.stp`. The drop handler regex matches the same set. These are consistent, which is good. However, the `accept` attribute is only a hint; the browser file picker may still allow other files. The drop handler re-filters, so this is safe. | None needed. |
| 3.2a | Run button disabled during slicing (single file) | UX Friction | OK | 2183, 2238 | `btn.disabled = true` is set at the start of `runQuote()` and `_doSingleQuote()`. Re-enabled at line 2267 in the finally path. Double-click is prevented. | None needed. |
| 3.2b | Run button disabled during batch slicing | UX Friction | OK | 2274, 2387, 2420 | `btn.disabled = true` at the start of `runBatch()` and `_runBatchSlice()`. Re-enabled at line 2420. | None needed. |
| 3.2c | Progress polling silently swallows errors | UX Friction | Low | 2013-2014 | In `startProgressPoll()`, the `catch(e) {}` block silently ignores all fetch errors. If the server goes down mid-slice, the progress panel just freezes with the last known state -- no indication that polling failed. | After N consecutive poll failures, show a "connection lost" indicator in the progress panel or stop polling with a message. |
| 3.2d | Progress panel always hidden initially, shown on demand | UX Friction | OK | 1456, 1988-2000 | `#progress-panel` starts `display:none` and is shown by `showProgressPanel()` which also hides other panels. Clean transitions. | None needed. |
| 3.3a | Single-file fetch errors shown to user | UX Friction | OK | 2255-2265 | Both HTTP error responses and network exceptions call `showError()`, which displays the `#error-box` div. The error box uses monospace font and `word-break: break-all` for readability. | None needed. |
| 3.3b | Batch errors shown per-row in table | UX Friction | OK | 2409-2414, 2433-2434 | Failed batch items get `status: 'error'` and the error message is shown in the batch table row with `batch-err` class (red text). Clicking the row opens the modal with details. | None needed. |
| 3.3c | Slicer stderr not directly surfaced to user | UX Friction | Medium | 2256-2257 | The `/api/quote` endpoint returns `data.error` which may contain slicer stderr, but it is presented as a raw string in the error box. There is no parsing or user-friendly formatting of common PrusaSlicer errors (e.g., "model is not manifold", missing profiles). | Parse common slicer error patterns and show user-friendly messages with the raw error available in an expandable section. |
| 3.3d | Init failure shows "server unreachable" in printer dropdown | UX Friction | Medium | 1808-1811 | If `/api/printers` fails, the printer dropdown shows "-- server unreachable --" but there is no prominent error banner. The user might not notice this subtle text in a dropdown they haven't opened. | Show a visible error banner at the top of the page when the server is unreachable at init time. |
| 3.3e | Pre-flight check failure falls through silently | UX Friction | Medium | 2226-2230 | If the pre-flight `/api/check_size` fetch throws, the error is caught, a progress step is marked 'error', then `hideProgressPanel()` is called and execution falls through to `_doSingleQuote()`. The user never sees the pre-flight error -- it is hidden and slicing proceeds. This is arguably intentional (best-effort pre-flight), but a console warning is the only trace. | Consider showing a brief warning that pre-flight check failed before proceeding, or keep current behavior but document it. |
| 3.4a | Button text updates when queue changes | UX Friction | OK | 2125, 2153, 1967 | `addFilesToQueue()` sets text to "Quote All" or "Generate Quote" based on queue length. `clearQueue()` resets to "Generate Quote". `deleteQueueItem()` also updates. | None needed. |
| 3.4b | Button disabled when no file loaded | UX Friction | OK | 1688, 1889 | The button starts `disabled` with `title="Upload a model file first"`. `updateRunBtn()` checks both `selectedFile` and printer selection. | None needed. |
| 3.4c | Button re-enables after single quote completion or error | UX Friction | OK | 2267-2268 | After `_doSingleQuote()` completes (success or catch), the button is re-enabled and text reset to "Generate Quote". | None needed. |
| 3.4d | Button re-enables after batch completion | UX Friction | OK | 2420-2421 | After `_runBatchSlice()` loop finishes, the button is re-enabled with text "Quote All". | None needed. |
| 3.4e | Button not disabled during pre-flight when user must choose | UX Friction | OK | 2221-2222, 2323-2324 | When pre-flight finds oversized files and shows the preflight panel, the button is re-enabled so the user can interact. The preflight panel has its own "Continue & Quote" button. | None needed. |
| 3.5a | Preflight panel hidden when results shown | UX Friction | OK | 1996, 2188, 2278 | `showProgressPanel()` hides the preflight panel. `runQuote()` and `runBatch()` also explicitly hide it at the start. | None needed. |
| 3.5b | Stale results panel hidden before new quote | UX Friction | OK | 2185-2189, 2275-2280 | Both `runQuote()` and `runBatch()` hide `#results`, `#batch-results`, `#placeholder`, `#preflight-panel`, and `#error-box` at the start. No stale content flicker. | None needed. |
| 3.5c | Placeholder not restored on clear | UX Friction | Low | 2147-2157 | `clearQueue()` does not explicitly restore the placeholder panel. After clearing, neither placeholder nor results are visible -- the right panel appears empty. | In `clearQueue()`, set `placeholder.style.display = 'flex'` and hide `#results`, `#batch-results`, `#error-box`, and `#rc-tabs`. |
| 3.6a | copyQuote includes all relevant fields | UX Friction | OK | 2664-2697 | `copyQuote()` includes filename, printer, profile, filament, orient note, split count, time, weight, volume, material/machine costs, markup, quote price, farm info, quantity totals, and per-piece breakdown. Comprehensive. | None needed. |
| 3.6b | copyQuote provides visual feedback | UX Friction | OK | 2693-2697 | After successful clipboard write, the button text changes to "Copied!" for 1.5 seconds. | None needed. |
| 3.6c | copyQuote has no error feedback | UX Friction | Low | 2693 | The `.then()` handles success but there is no `.catch()` handler (the promise chain does not surface clipboard errors). If clipboard access is denied (e.g., in non-HTTPS contexts), nothing happens -- no error shown. | Add a `.catch(() => { btn.textContent = 'Copy failed'; setTimeout(...) })` handler. |
| 3.6d | copyModalQuote has no user feedback | UX Friction | Medium | 2501-2513 | `copyModalQuote()` calls `navigator.clipboard.writeText(...).catch(()=>{})` but provides no visual confirmation (no "Copied!" flash). The user has no idea if the copy succeeded. | Add the same "Copied!" flash pattern used in `copyQuote()`. |
| 3.7a | Modal closes on Escape key | UX Friction | OK | 2862 | `document.addEventListener('keydown', e => { if (e.key === 'Escape') closeQuoteModal(); })` -- works correctly. | None needed. |
| 3.7b | Modal closes on backdrop click | UX Friction | OK | 2879 | `onclick="if(event.target===this)closeQuoteModal()"` on `#quote-modal` -- clicking the backdrop (not the inner content) closes the modal. | None needed. |
| 3.7c | Body scroll not prevented when modal open | UX Friction | Low | 2496, 2498-2500 | `openQuoteModal()` adds the `open` class but does not set `document.body.style.overflow = 'hidden'`. The background page can scroll while the modal is open. On short pages this is negligible, but on long form panels the user could accidentally scroll behind the modal. | Set `body.style.overflow = 'hidden'` on open, restore on close. |
| 3.7d | Multiple modals cannot be opened | UX Friction | OK | 2454-2497 | There is only one modal element (`#quote-modal`). `openQuoteModal()` always populates and shows the same element, effectively replacing any previous content. No stacking issue. | None needed. |
| 3.8a | Single responsive breakpoint at 800px | UX Friction | Low | 1280 | The only `@media` query is `@media (max-width: 800px) { .layout { grid-template-columns: 1fr; } }`. This collapses the two-column grid to single-column. No elements are hidden -- good. | Consider additional breakpoints for very small screens (< 480px) where padding and font sizes could be adjusted. |
| 3.8b | Inputs use width:100% -- responsive-safe | UX Friction | OK | 1308 | All `select` and `input` elements use `width: 100%`, so they naturally fill their container in single-column layout. | None needed. |
| 3.8c | Results card stacks below controls on narrow screens | UX Friction | OK | 1280, 1272-1278 | The grid collapses to `1fr`, putting the results card below the controls card. `align-items: start` ensures no awkward stretching. | None needed. |
| 3.8d | Debug panel fixed bottom, full width -- works on mobile | UX Friction | OK | 1495 | `position:fixed; bottom:0; left:0; right:0;` -- the debug panel spans the full width regardless of viewport. `max-height:55vh; overflow-y:auto` keeps it from consuming the entire screen. | None needed. |
| 3.9a | Infill has min/max/step | UX Friction | OK | 1630 | `min="0" max="100" step="5"`. Browser native validation applies on number input. | None needed. |
| 3.9b | Layer height has min/max/step | UX Friction | OK | 1634 | `min="0.05" max="1.0" step="0.01"`. Reasonable constraints. | None needed. |
| 3.9c | Quantity allows zero | UX Friction | Medium | 1679 | `<input type="number" id="quantity" value="1" min="1" step="1">`. The HTML `min="1"` provides browser-level validation, but there is no JS-side validation. A user can type `0` or `-1` and submit. The server would process `quantity=0`, resulting in a zero-cost quote, or negative quantity with undefined behavior. | Add client-side validation in `getFormSettings()` or `buildFormData()` to clamp quantity to >= 1. Also validate server-side. |
| 3.9d | Cost per kg allows zero but not negative (via min=0) | UX Friction | OK | 1665 | `min="0"` prevents negative values via browser UI (spinner), but a user can still type a negative number. No JS validation. | Add JS validation or server-side clamping. Low severity since `min="0"` handles the common case. |
| 3.9e | Markup has no max -- allows extreme values | UX Friction | Low | 1675 | `min="0"` but no `max`. A user could enter markup of 99999%. Not harmful but could produce confusingly large quotes. | Consider adding a reasonable `max` (e.g., 500%) or a confirmation for extreme values. |
| 3.9f | No client-side validation before submit | UX Friction | Medium | 2177-2234 | `runQuote()` checks only `selectedFile` and printer selection. No validation of numeric inputs (cost_per_kg, hourly_rate, markup, quantity, farm_size). All values are sent as strings to the server. If the user leaves pricing fields blank, the quote returns without cost breakdown (handled gracefully on server), but there is no warning. | Add a lightweight validation pass before submitting: warn if pricing fields are empty, clamp out-of-range values. |
| 3.9g | Farm size allows zero | UX Friction | Low | 1685 | `min="1"` in HTML but no JS validation. Typing `0` would cause a division-by-zero on the server when calculating farm time. | Validate farm_size >= 1 client-side and server-side. |
| 3.10a | Toggle buttons have clear active state | UX Friction | OK | 1319-1324 | `.toggle-btn.active` gets `background: var(--accent); border-color: var(--accent); color: #fff;` which is visually distinct from the inactive state (dark background, muted text). High contrast difference. | None needed. |
| 3.10b | Only one toggle active at a time | UX Friction | OK | 1833-1853 | `setSizeMode()` and `setSupportMode()` toggle the `active` class exclusively -- only the selected button gets it. No multi-select possible. | None needed. |
| 3.11a | Debug panel overlays content (fixed position) | UX Friction | OK | 1495 | `position:fixed; bottom:0;` means the debug panel overlays the main UI rather than pushing it. The main UI remains interactive above the panel. | None needed. |
| 3.11b | Debug button position could conflict with content | UX Friction | Low | 1492 | The debug button is `position:fixed; bottom:1.2rem; right:1.2rem; z-index:900`. On narrow screens, this could overlap the bottom of the results card or the run button. | Consider hiding or repositioning the debug button on very small viewports, or making it semi-transparent until hovered. |
| 3.11c | Debug panel has timestamps and tracebacks | UX Friction | OK | 2927-2938 | Each entry shows timestamp, endpoint, error type/message, and an expandable traceback. Entries are clickable to expand/collapse. Useful for debugging. | None needed. |
| 3.11d | Debug panel polls every 10 seconds for new errors | UX Friction | Low | 2962-2970 | A `setInterval` runs every 10 seconds to check for new errors, even when the panel is closed. This updates the button badge count. The polling is lightweight but adds constant network traffic. | Consider using a longer interval (30s) or stopping polling when the tab is not visible (use `document.visibilityState`). |
| 3.11e | Main UI fully interactive while debug panel is open | UX Friction | OK | 1495 | The debug panel uses `max-height:55vh` so the top portion of the page remains visible and interactive. The panel has its own scroll. | None needed. |
| 3.12a | Empty preset name prevented | UX Friction | OK | 2777-2778 | `confirmSavePreset()` trims the name and returns early (refocusing the input) if empty. | None needed. |
| 3.12b | Duplicate preset name silently overwrites | UX Friction | Medium | 2776-2803 | If a user saves a preset with a name that already exists, the server endpoint overwrites it without warning. | 3.12c | Delete has confirmation dialog | UX Friction | OK | 2805-2826 | `deletePreset()` shows a confirmation row with the preset name ("Delete 'X'?") with Delete/Cancel buttons. The preset bar is hidden during confirmation to prevent confusion. | None needed. |
| 3.12d | Delete with no preset selected is a no-op | UX Friction | OK | 2806-2807 | `deletePreset()` returns early if no preset is selected (`if (!name) return`). | None needed. |
| 3.12e | Preset dropdown clearly labeled | UX Friction | OK | 1537 | The dropdown default option is "-- saved presets --" which is clear. The section has a "Presets" card title above it. | None needed. |
| 3.12f | Preset Enter key shortcut works | UX Friction | OK | 2858-2861 | The preset name input listens for Enter (save) and Escape (cancel) keydown events. | None needed. |
| 3.12g | Load preset with no selection is a no-op | UX Friction | OK | 2743-2744 | `loadPreset()` returns early if no name is selected or the name is not in cache. | None needed. |
| 3.12h | Preset size_mode and size_val not saved | UX Friction | Medium | 2779-2792 | The `confirmSavePreset()` body includes printer, profiles, infill, layer_height, walls, supports, cost_per_kg, hourly_rate, markup, and farm_size.
