| 5.4a | Support mode values not validated on server side | Bug | Medium | 541-552, 863, 1848-1854 | JS sends one of `'none'`, `'grid'`, `'snug'`, `'organic'` (line 1850). Python receives via `form.get("supports", "")` (line 863) and passes directly to `_override_flags()` (line 541). No validation occurs. If a crafted request sends `supports=foo`, PrusaSlicer receives `--support-material-style foo` and will either silently ignore it, use a default, or fail with an opaque error like "Unknown support style". The user sees a raw slicer stderr dump. | Add validation: `if supports not in ('none', 'grid', 'snug', 'organic', ''): return jsonify({"error": "Invalid support mode"}), 400`. |
| 5.4b | Other override values also unvalidated | Bug | Low | 541-552, 860-862 | `infill`, `layer_h`, and `walls` are passed directly to PrusaSlicer CLI flags without validation. Malformed values (e.g., `infill=abc`, `layer_h=-1`, `walls=0`) will produce unhelpful slicer errors. However, these are less likely to be accidentally triggered via the UI since the inputs are numeric fields. | Add basic server-side validation (numeric check, range check) for `infill`, `layer_height`, `walls`. |
| 5.5a | INI files without section headers: lines silently ignored | Edge Case | Low | 147-163 | `_load_ini_sections()` only processes key=value lines when `current` is set (line 158: `elif current and "=" in line`). Lines before any `[section]` header are silently dropped. PrusaSlicer user INI files typically have a `# generated by PrusaSlicer` comment header followed by key=value pairs with NO section header. These files are handled separately by `_patch_machine_limits()` and `build_machine_limits_ini()` which have their own flat parsers (lines 566-577, 614-621). The `_load_ini_sections()` function is only used for bundled vendor profiles which always have section headers. | No fix needed -- the architecture correctly uses different parsers for different file formats. Document this distinction. |
| 5.5b | Keys with `=` in values: only first `=` used as delimiter | OK | None | 158-160 | `line.partition("=")` correctly splits on the first `=` only, so values containing `=` are preserved correctly (e.g., `gcode_flavor = reprap;firmware_retraction=1`). | None needed. |
| 5.5c | Deep inheritance chains: capped at depth 8 | OK | None | 179-181 | `_resolve_key()` has `if depth > 8: return None`, preventing infinite recursion. The function tries 4 name variants per parent (line 192-197), so each level fans out 4x, giving a worst-case of 4^8 = 65,536 calls. In practice, inheritance chains rarely exceed 2-3 levels, and there are typically only a few dozen sections in `_all_printer_sections()`. | None needed in practice, but could add memoization if profile databases grow large. |
| 5.5d | Circular `inherits` references: not explicitly detected | Bug | Low | 179-201 | `_resolve_key()` does not track visited sections. If profile A inherits from B and B inherits from A, the function will recurse until `depth > 8` and return `None`. It will NOT crash (depth cap prevents stack overflow), but it will silently fail to find the key even if it exists in one of the profiles. With the 4 candidate variants per level, this creates exponential redundant work (up to 65K calls) before the depth cap kicks in. | Add a `visited` set parameter to skip already-visited sections. This makes circular references fail fast and prevents redundant traversal. |
| 5.5e | `_all_printer_sections()` overwrites duplicate section names | Edge Case | Low | 166-176 | When the same section name `[printer:Name]` appears in both a vendor INI and a user INI, the user version overwrites the vendor version (line 175: `all_sections[sec] = kv`). This is correct behavior (user profiles should take priority). However, if the SAME section appears in multiple vendor INI files, the last one loaded wins, which depends on glob ordering. | Document that user profiles take priority. Glob ordering for vendor files is filesystem-dependent but unlikely to cause issues since vendor section names are unique. |
